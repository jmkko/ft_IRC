53 * @brief check business validity of args before adjusting modes
54 * @details
55 * - supported modes are channel modes
56 - `k` key
57 - `i` invite-only
58 - `l` member limit
59 - `t` restrict usage of TOPIC to operators
60 and user mode `o` : make operator
61 * - supported operations are `+` and `-`
62 * - modes requiring arguments are `k` (valid key), `l` (int from -1 to INT_MAX) abd `o` (valid username)
63 * sends a RPL_CHANNELMODEIS in case it is successful - and RPL_YOUREOPER (for +o)
64 * @remark parsing is less flexible than RFC suggests : if a mode requires a value, it should be provided after the modes at
65 the corresponding index. In other words, `+kl key 10` is accepted but `+k key +l 10` is rejected
66 * @remark If duplicate modes are present (as in `+kk key1 key2`) the last one prevails
67 * @param server
68 * @param client
